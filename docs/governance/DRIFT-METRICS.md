## Known-Acceptable Conditions (by design)

### 1. Trace vault has only 2 seals
**Condition:** `specsmith trace log` shows 2 seals (SEAL-0001 and SEAL-0002).
**Why acceptable:** The trace vault was started after the initial AEE lifecycle
establishment (SEAL-0001 from Phase 1 release). Phase 2 and Phase 3 work
(dep graph, rollback, context pack, query API, metrics, NDJSON WAL, PyO3
bindings) proceeded without new seals because those were implementation
cycles on already-accepted requirements, not new architectural decisions.
**Action required:** None. New seals should be added at future major phase
transitions or release milestones.

### 2. LEDGER.md is sparse (65 lines)
**Condition:** LEDGER.md contains fewer entries than a mature project would have.
**Why acceptable:** chronomemory was bootstrapped as a greenfield library
directly at `Release` phase. Many governance decisions were established
upstream in `specsmith` (which hosts the ESDB master spec) rather than
recorded in this repo's ledger.
**Action required:** Continue appending ledger entries for major decisions.
Current threshold (≤ 500 lines) is not a concern.

### 3. No scaffold.yml
**Condition:** `specsmith validate` reports "No scaffold.yml found".
**Why acceptable:** chronomemory predates the scaffold.yml convention and was
not created via `specsmith init`. The project is fully governed without it.
**Action required:** None. This is a persistent `std: passed` (non-error)
condition by design.

### 4. ARCHITECTURE.md references only 7 REQ IDs explicitly
**Condition:** `specsmith validate` reports `architecture.md references 7 REQ IDs`.
**Why acceptable:** ARCHITECTURE.md references requirements inline where
relevant (e.g., `REQ-CM-009`, `REQ-CM-015` through `REQ-CM-021`). The
remainder of the 24 REQs are covered by the module documentation sections
but without explicit `REQ-CM-NNN` strings in the text. This passes validation.
**Action required:** None. The check passes; additional inline references
would be added if arch sections are updated.

### 5. Rust backend (RUST_BACKEND = False) without compilation
**Condition:** `from chronomemory import RUST_BACKEND` returns `False` unless
the developer explicitly compiles the PyO3 extension:
`maturin develop --manifest-path crates/chronomemory-py/Cargo.toml`.
**Why acceptable:** The pure-Python implementation is complete, tested, and
the primary interface. The Rust backend (`_chronomemory_rust`) is an optional
performance accelerator — not required for correctness or governance.
**Action required:** None. Document in AGENTS.md and onboarding materials
that the Rust backend is opt-in.

### 6. Rust WAL format (NDJSON) is not yet cross-validated against Python in CI
**Condition:** CI runs Python tests and Rust tests independently. There is no
automatic cross-language WAL round-trip test in CI (only a local verification
was done manually: Python WAL fields confirmed to match NDJSON schema).
**Why acceptable:** The hash computation and field schema are verified in both
test suites separately. A dedicated cross-compat integration test
(`test_cross_wal_roundtrip_rust_python`) is tracked as Phase 4 work.
**Action required:** Open a GitHub issue for the cross-compat CI test.

### 7. docs/REQUIREMENTS.md and docs/TESTS.md are derived artifacts
**Condition:** These files are auto-generated by `specsmith sync` from the
YAML canonical sources (`docs/requirements/chronomemory.yml` and
`docs/tests/chronomemory.yml`). They must not be hand-edited.
**Why acceptable:** This is the intended YAML-first governance mode
(`governance-mode = yaml`). The generated Markdown is for human readability
only. All edits go to the `.yml` sources.
**Action required:** None. This is by design. Documented in ARCHITECTURE.md §31.
